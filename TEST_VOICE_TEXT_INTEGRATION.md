# 语音与文字互通测试指南

## 测试前准备

1. 确保后端服务运行：`cd Backed && npm start`
2. 确保前端服务运行：`cd Unified && npm run dev`
3. 确保 xiaozhi WebSocket 服务可访问
4. 确保 Dify API 配置正确

## 测试用例

### 测试 1：文字到语音的上下文传递

**目标**：验证 xiaozhi 能看到之前的文字对话

**步骤**：
1. 登录客户端账号
2. 在文字聊天界面输入："我今年65岁，有高血压"
3. 等待 AI 回复
4. 继续输入："我想制定一个运动计划"
5. 等待 AI 回复
6. 点击语音按钮进入语音模式
7. 说："刚才说的运动计划，具体应该怎么做？"

**预期结果**：
- xiaozhi 能理解"刚才说的运动计划"指的是之前文字对话中提到的内容
- xiaozhi 的回复会考虑用户的年龄（65岁）和健康状况（高血压）
- 控制台显示：`传递 X 条历史消息给 xiaozhi`
- 控制台显示：`开始录音会话，附带历史消息上下文 (X 条)`

**验证点**：
- [ ] 语音识别正常
- [ ] xiaozhi 理解上下文
- [ ] 回复内容连贯
- [ ] 控制台日志正确

---

### 测试 2：语音到文字的上下文传递

**目标**：验证 Dify 能看到之前的语音对话

**步骤**：
1. 登录客户端账号
2. 点击语音按钮进入语音模式
3. 说："我最近睡眠不好"
4. 等待 AI 回复
5. 说："经常半夜醒来"
6. 等待 AI 回复
7. 挂断电话，返回文字界面
8. 在文字输入框输入："除了刚才说的，我还有什么需要注意的？"

**预期结果**：
- Dify 能理解"刚才说的"指的是语音对话中的睡眠问题
- Dify 的回复会基于之前的语音对话内容
- 文字界面显示完整的对话历史（包括语音部分）

**验证点**：
- [ ] 语音对话记录显示在文字界面
- [ ] Dify 理解上下文
- [ ] 回复内容连贯
- [ ] 数据库中消息完整

---

### 测试 3：多次切换

**目标**：验证多次切换后上下文仍然正确

**步骤**：
1. 文字：输入"我想了解营养搭配"
2. 等待回复
3. 切换到语音：说"早餐应该吃什么"
4. 等待回复
5. 切换到文字：输入"午餐呢"
6. 等待回复
7. 切换到语音：说"晚餐有什么建议"
8. 等待回复

**预期结果**：
- 每次切换后，AI 都能理解之前的对话
- "午餐呢"能被理解为"午餐应该吃什么"
- "晚餐有什么建议"能被理解为营养搭配的一部分

**验证点**：
- [ ] 4 次切换都成功
- [ ] 每次 AI 回复都有上下文
- [ ] 消息顺序正确
- [ ] 没有消息丢失

- [ ] 技能识别正确

### 测试 6：会话恢复

**预期结果**：
- 刷新后历史消息完整显示
- 继续对话时，AI 能理解之前的内容
- `conversationId` 保持不变

**验证点**：
- [ ] 历史消息加载完整
- [ ] conversationId 正确恢复
- [ ] 上下文连贯

### 测试 7：新对话创建

**目标**：验证新对话时正确处理

**步骤**：
1. 点击"新对话"按钮
2. 在文字界面输入消息
3. 等待 Dify 返回新的 `conversation_id`
4. 切换到语音模式
5. 说话

**预期结果**：
- 新对话创建成功
- 获得新的 `conversation_id`
- 语音模式使用新的 `conversation_id`
- 历史消息为空（只有当前对话）

**验证点**：
- [ ] 新 conversationId 生成
- [ ] 保存到 localStorage
- [ ] 语音模式使用正确的 ID
- [ ] 不会混入旧对话

## 调试技巧

### 查看控制台日志

**前端日志**：
```
传递 X 条历史消息给 xiaozhi
开始录音会话，附带历史消息上下文 (X 条)
历史上下文预览: 用户: xxx\nAI: xxx...
```

**后端日志**：
```
[AI顾问聊天-客户端-dify-流式] 用户 X 发送消息
  对话ID: xxx
  消息: xxx
```

### 检查数据库

```sql
-- 查看某个会话的所有消息
SELECT id, role, content, timestamp 
FROM ai_chat_messages 
WHERE conversation_id = 'xxx' 
ORDER BY timestamp ASC;

-- 查看最近的消息
SELECT * FROM ai_chat_messages 
ORDER BY timestamp DESC 
LIMIT 20;
```

### 检查 WebSocket 消息

在浏览器开发者工具的 Network 标签中：
1. 筛选 WS (WebSocket)
2. 查看发送的消息
3. 验证 `context` 字段是否包含历史消息

### 检查 localStorage

```javascript
// 在浏览器控制台执行
console.log(localStorage.getItem('current_conversation_id'))
```

## 常见问题

### Q1: xiaozhi 没有理解之前的文字对话
**可能原因**：
- 历史消息没有正确传递
- WebSocket 连接问题
- xiaozhi 服务端不支持 `context` 字段

**解决方法**：
- 检查控制台日志，确认历史消息已传递
- 检查 WebSocket 消息格式
- 联系 xiaozhi 服务端确认 API 支持

### Q2: Dify 没有理解之前的语音对话
**可能原因**：
- `conversation_id` 不一致
- 消息没有保存到数据库
- Dify API 配置错误

**解决方法**：
- 检查数据库中的消息记录
- 验证 `conversation_id` 是否一致
- 检查 Dify API 响应

### Q3: 切换模式后消息丢失
**可能原因**：
- 前端状态管理问题
- 数据库保存失败
- 网络请求失败

**解决方法**：
- 检查浏览器控制台错误
- 检查网络请求状态
- 验证数据库写入

### Q4: 技能卡片不显示
**可能原因**：
- 技能识别失败
- 前端渲染问题
- 数据格式错误

**解决方法**：
- 检查技能关键词匹配
- 验证 `skillSteps` 数据结构
- 查看前端渲染逻辑

## 性能测试

### 测试场景
1. 100 条消息的长对话
2. 快速切换模式（10 次/分钟）
3. 多个用户同时使用
4. 网络延迟模拟

### 性能指标
- 消息发送延迟 < 500ms
- 模式切换时间 < 1s
- 历史消息加载 < 2s
- 内存占用稳定
